# 规则仓库审计与排错记录

本文记录规则仓库和 App 联合使用时必须检查的点，后续出问题先按本文排查。

## 当前结论

当前规则仓库已经不是空规则包。最终给 App 读取的是：

```text
生成规则：52+ 条（随运行持续增长）
手工稳定规则：3 条
总规则：55+ 条
```

App 默认读取地址：

```text
https://raw.githubusercontent.com/liyan-lucky/ComicReader_Rules/main/generated/index.json
```

只要这个文件里的 `rules` 不是空数组，App 在设置页点"从 GitHub 更新规则"后就能加载远程规则。

## 仓库分工

```text
ComicReader_HarmonyOS：App 工程仓库。
ComicReader_Rules：远程规则仓库。
```

App 不读取发布页附件，也不读取压缩包。App 默认只读取主分支里的：

```text
generated/index.json
```

发布页和压缩包主要给人下载检查，不能代替 `generated/index.json`。

## 规则生成流程

当前流程：

```text
运行工作流
→ 种子发现（从29个漫画站抓取站内链接）
→ Brave API 搜索（宽泛查询，1条覆盖多域名）
→ DuckDuckGo HTML 兜底（连续403自动跳过）
→ 候选去重 + 前置过滤（屏蔽非漫画域名）
→ 审计候选（检测登录/付费早退出，省HTTP请求）
→ build_index_from_report.py 生成索引
→ sanitize_rule_outputs.py 清洗非漫画源 + 重写 .ets
→ update_manifest.py 更新版本入口
→ 合并手工稳定规则
→ 写入 generated/index.json
→ 同步 rules/index.json
→ 提交到主分支
→ 打标签
→ 发布附件
```

其中最关键的是"合并手工稳定规则"。自动搜索可能因为搜索接口限制、站点结构变化、公开页面不合格而生成 0 条规则。如果没有手工稳定规则兜底，App 远程更新后就会显示 0 条。

## 已处理过的问题

### 1. 工作流重复

仓库里曾经同时存在两个工作流，名字相同，容易点错。旧工作流只生成文件，不发布；新版工作流才是一键运行并发布。现在旧工作流已删除，只保留新版。

### 2. 推送被拒绝

报错内容类似：

```text
推送被拒绝，需要先拉取远程主分支
```

原因是工作流运行期间，主分支被其它提交更新了。已处理：工作流提交前会自动拉取远程主分支并重新整理提交，推送失败会重试，并且加了并发锁，避免多个规则流程互相抢推送。

### 3. 发布附件重名

报错内容类似：

```text
附件名称已存在
```

原因是同时上传 `generated/index.json` 和 `rules/index.json` 时，发布页里都会变成同名的 `index.json`。已处理：发布附件改为唯一名称：

```text
generated-index.json
rules-index.json
rulebot-report.json
GeneratedSourceRules.ets
```

### 4. 流程成功但规则数量为 0

流程成功只能说明脚本运行完成，不代表 App 有可用规则。必须检查：

```text
generated/index.json 里的 rules 数组数量
audit.totalCount
```

现在已经加入 `rules/manual/index.json`，自动规则为 0 时也会把手工稳定规则放进最终索引。

### 5. build_index_from_report.py NameError: safe_str 未定义

**时间**：2026-07-06

**现象**：workflow "生成公开规则"步骤报错 `NameError: name 'safe_str' is not defined`

**原因**：`build_index_from_report.py` 第76行调用了 `safe_str()` 函数，但该函数只在 `sanitize_rule_outputs.py` 中定义，未在 `build_index_from_report.py` 中定义。

**修复**：在 `build_index_from_report.py` 中添加 `safe_str()` 函数定义。

### 6. 搜索全部403，种子发现被时间预算挤掉

**时间**：2026-07-06

**现象**：quick 模式运行30分钟后生成0条规则，`rawCandidateCount: 0`

**原因**：
- DuckDuckGo HTML 搜索全部返回403（被反爬拦截）
- Brave/Google API 无密钥跳过
- 400条搜索查询耗尽30分钟时间预算，种子发现还没开始就被截断

**修复**：
1. 种子发现移到搜索之前执行（确保种子候选不被时间预算挤掉）
2. DuckDuckGo 连续403五次后快速跳过剩余搜索查询
3. 添加 `cloudscraper` 绕过 Cloudflare
4. 修复 User-Agent（去掉 `ComicRuleBot` 标识）

### 7. 种子站太少，URL过滤过严

**时间**：2026-07-06

**现象**：只有3个域名（kaixinman.com, mgeko.cc, comick.io）返回候选，其余全部 `fetch_failed`

**原因**：
- 种子站只有7个域名
- `likely_seed_candidate_url` 对同域链接过滤过严
- 大部分漫画站被 Cloudflare 拦截

**修复**：
1. 种子站从7个扩展到29个域名
2. 放宽种子候选URL过滤（对同域链接更宽松）
3. 给已知可工作站增加深层分类页种子
4. 提高每域名规则上限（5→20→100）

### 8. 效率重构

**时间**：2026-07-07

**现象**：Brave API 配额浪费、审计请求浪费、.ets 文件双重生成

**修复**：
1. 前置 sanitize 屏蔽列表到候选过滤阶段（抖音/b站/微博等不再浪费审计请求）
2. 移除 generate_rules.py 的 .ets 输出（被 sanitize 完全覆盖，纯浪费）
3. 统一规则ID后缀（`_remote_public` → `_auto_public`）
4. 有种子发现的域名跳过 `site:` 搜索查询
5. 检测到登录/付费且无章节时跳过章节页请求
6. Brave API 可用时用宽泛查询（不用 `site:` 前缀）
7. 本地脚本补全 sanitize + manifest 步骤

## 每次发布后检查

发布后先看这三个位置：

```text
generated/index.json
rules/index.json
发布页附件
```

重点看：

```text
rules 是否为空
audit.generatedCount 是多少
audit.manualCount 是多少
audit.totalCount 是多少
```

只要 `totalCount` 大于 0，App 才有远程规则可加载。

## App 侧使用方法

App 内进入：

```text
设置 → GitHub 远程规则 → 从 GitHub 更新规则
```

看到"远程规则更新成功：X 条"后，再回搜索页搜索漫画名。

如果远程规则条数为 0，App 仍会使用内置本地规则，但远程规则仓库本身就没有发挥作用。

## 不做的事情

规则仓库只处理公开网页和公开图片地址，不做这些事情：

```text
不登录
不付费
不绕验证码
不破解加密接口
不伪造专用客户端协议
不做反爬绕过
```

如果某站点必须登录、必须付费、必须客户端解密，规则仓库不会生成它的可用图片规则。
